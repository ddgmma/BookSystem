## 📍 做项目的正确路径（金字塔模型）

```
          ▲
         / \
        / 5 \   ← 优化重构
       /     \
      /   4   \  ← 加数据库/持久化
     /         \
    /     3     \ ← 写业务逻辑
   /             \
  /       2       \ ← 先跑通核心功能（内存版）
 /                 \
/________ 1 ________\ ← 明确需求、建模型
```

---

## 🧭 详细拆解每个台阶

### 第 1 层：想清楚 → 写纸上（最重要的一步！）

> **用自然语言描述：这个系统到底是干什么的？**

**错误示范：** 打开 IDEA 直接开始写 `public class`... 然后卡住。

**正确做法：** 拿纸笔，写两样东西：

**① 功能清单（用户视角）**

```
图书管理系统能做什么？
- 管理员可以添加一本新书（书名、作者、库存数量）
- 管理员可以查看所有图书
- 管理员可以搜索图书（按书名模糊搜索）
- 用户可以借书（选择图书 + 输入姓名）
- 用户可以还书（输入图书编号）
```

**② 数据模型（核心实体）**

```
Book（图书）
  - id
  - title（书名）
  - author（作者）
  - stock（库存数量）

BorrowRecord（借阅记录）
  - id
  - bookId（借了哪本书）
  - borrower（谁借的）
  - borrowDate（什么时候借的）
```

> ✅ **判断标准：** 一个完全不懂编程的人，看了你的清单，也能说清楚这个系统是干什么的。

---

### 第 2 层：搭骨架 → 最小可运行版本

> **不要一次性实现所有功能！先实现一个"能跑通的最小闭环"。**

**所谓最小闭环：** 用户能完成一个完整的操作流程。

```
【最小闭环示例：添加图书 → 查看图书】
1. 程序启动，显示菜单
2. 选"1"，输入书名
3. 选"2"，看到刚才添加的书
4. 选"0"，退出
```

**这时候你的代码结构：**

```
只用两个文件：
├── Book.java          ← 模型类
└── Main.java          ← 一个main方法 + 一个ArrayList + 两个if分支
```

**核心原则：**

- ❌ 不要管什么分层架构、设计模式
- ❌ 不要管数据库
- ✅ **怎么简单怎么来，先让代码跑起来！**

> 💡 这步的**唯一目标**：让你看到"输出"（`System.out.println`），获得正反馈。

---

### 第 3 层：加功能 → 逐步完善业务逻辑

此时你已经有一个"跑得通"的版本了，开始**一个一个地加功能**：

```
版本 1.0 → 添加图书 ✓
版本 1.1 → 查看所有图书 ✓
版本 1.2 → 搜索图书 ✓
版本 1.3 → 借书功能 ✓
版本 1.4 → 还书功能 ✓
版本 1.5 → 借书记录查询 ✓
```

**每一步只加一个功能，加完就运行测试一下！**

**什么时候开始做分层？** 当你的 `Main.java` 超过 200 行，或者你发现一个方法里有三种不同的事情混在一起时。

```
这时才拆分：
├── model/          ← 数据模型
├── service/        ← 业务逻辑
└── Main.java       ← 只负责界面
```

> ✅ **判断标准：** 你的系统所有基本功能都能正常工作，不依赖数据库，数据存在内存里。

---

### 第 4 层：上数据库 → 数据持久化

> **为什么这一步不是第 1 步？**

因为如果你一开始就搞 JDBC，你会在**两件事上同时遇到问题**：

1. 业务逻辑有没有写对？
2. 数据库连接/SQL 有没有写对？

**你根本无法定位 bug 出在哪里！**

**正确做法：**

1. 先保证**内存版**的所有功能都正常
2. 只替换"数据存储"那一层（Repository/DAO）
3. 其他代码（业务逻辑、界面）**一行不改**

```java
// 第一步：内存版
class BookRepository {
    private List<Book> books = new ArrayList<>();
  
    public void save(Book book) { books.add(book); }
    public List<Book> findAll() { return new ArrayList<>(books); }
}

// 第二步：换成JDBC版（接口保持不变）
class BookRepository {
    private Connection conn = DriverManager.getConnection(...);
  
    public void save(Book book) {
        // 执行 INSERT SQL
    }
    public List<Book> findAll() {
        // 执行 SELECT SQL，组装成 Book 对象
    }
}
```

> ✅ **判断标准：** 程序重启后，之前添加的数据仍然存在。

---

### 第 5 层：优化重构 → 让代码更优雅

**这一步是锦上添花，不是必须的！** 你完全可以先去学 Spring Boot，回头再来优化。

可能的优化方向：

- 用 `Stream` 替代 `for` 循环
- 用异常处理替代 `if-else` 判断
- 用接口实现多态（比如 BookRepository 接口，方便切换实现）
- 用自定义注解 + 反射实现一个极简 ORM

## 🎯 用一个实际的例子说明这个路径

### ❌ 错误路径（很多人掉的坑）

```
Day 1:  设计 6 个表，画 ER 图 → 卡住
Day 2:  配置 Maven + JDBC 连接池 → 报错，卡两天
Day 3:  开始写代码，发现不会处理日期 → 又卡住
Day 4:  放弃 😢
```

### ✅ 正确路径

```
Day 1:  Book + Main，控制台输入"书名"，存到 ArrayList，打印出来 → 跑通！🎉
Day 2:  加"查看所有图书"功能（迭代版本）
Day 3:  加"搜索图书"功能（Stream API 练手）
Day 4:  加"借书/还书"功能（业务逻辑）
Day 5:  把 ArrayList 换成 JDBC → 数据持久化
Day 6:  代码分层（model/service/repository）
Day 7:  加异常处理 + 小优化
        → 成就感满满！💪
```

## 📐 一张图总结项目心法

```
┌─────────────────────────────────────────┐
│           正确路径 = 迭代开发             │
├─────────────────────────────────────────┤
│                                         │
│  第1步：先跑通一个功能（哪怕代码很丑）      │
│        ↓                                │
│  第2步：再加一个功能                      │
│        ↓                                │
│  第3步：再加一个功能                      │
│        ↓                                │
│  第4步：重构（加数据库/分层）              │
│        ↓                                │
│  第5步：继续加功能或学习新框架             │
│                                         │
└─────────────────────────────────────────┘
```

**记住三个"不要"：**

1. ❌ **不要一开始就追求完美架构**（先让代码跑起来）
2. ❌ **不要一次想做太多功能**（一次一个，逐步迭代）
3. ❌ **不要卡在一个问题上超过 30 分钟**（先绕过，用最简单的方式实现）

**三个"要"：**

1. ✅ **要频繁运行测试**（每写 10 行代码就运行一次）
2. ✅ **要先实现最小闭环**（让用户能完成一个完整操作）
3. ✅ **要保持快乐**（做项目是为了获得成就感，不是为了折磨自己 😄）
